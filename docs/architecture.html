Architecture

SpecTracer is a lean Python CLI with no heavy infrastructure. A handful of modules orchestrate a simple pipeline: collect → parse → link → aggregate → render (HTML and, optionally, JSON).

Modules

📄

models.py

Core data classes: Scenario, TestResult, RequiredLayer, ScenarioView plus helper functions. 90 lines — the entire type system.

🔍

parsers.py

Three parsers: Gherkin (line-by-line), JUnit XML (ElementTree), Cucumber JSON (json module). Results carry an optional module key from config.

🔗

linker.py

OR-based tag matching. Links scenarios to test results by shared tags. 17 lines — the simplest module.

📊

aggregator.py

Computes completion stats, layer breakdowns, feature trees, health checks, and unlinked results. 223 lines of pure logic.

🗃

collectors.py

File discovery: resolves feature file, JUnit XML, and Cucumber JSON paths with recursive directory scanning. 44 lines.

💻

renderers.py

Jinja2 template with a few lines of embedded HTML/CSS/JS. Produces the self-contained report file.

📋

report_model.py

Builds the JSON report from the same views/stats/health data the HTML renderer consumes — one data model, two outputs, no drift. Emitted only when output_json is set.

⚙

cli.py

Orchestrator: loads config, collects files, runs pipeline, writes HTML and (optionally) JSON output.

Parsers

FeatureParser

Reads .feature files line-by-line. No heavy Gherkin parser dependency — just enough to extract:

  • Feature: lines for the feature name
  • Scenario names (line following Scenario:)
  • Tags on lines above scenarios (distinguishes linking tags from @require-* tags)
  • Given/When/Then steps for display in the report

Default behavior: Scenarios without any @require-* tags default to @require-e2e.

JunitParser

Parses JUnit XML using xml.etree.ElementTree. Reads tags from three locations to support different frameworks:

  • name attribute on <testcase>
  • classname attribute on <testcase>
  • <properties><property> elements inside <testcase>

Handles <failure>, <error>, and <skipped> child elements for status classification. The config module key (e.g. "auth" under unit) is stamped onto each result as module.

CucumberParser

Parses Cucumber JSON output. Handles both array-of-scenarios and single-object formats. Extracts:

  • Scenario-level tags array for linking
  • Scenario name and status
  • Duration (handles nanosecond-to-second conversion)
  • Module scope from the config key under e2e (same model as unit/integration)

Linker

The linker is the simplest but most important module. It performs OR-based tag matching:

  • For each scenario, find all test results where any tag appears in the scenario's linking tags
  • @require-* tags are stripped before linking — they never participate
  • Tag matching is exact string equality only — no wildcards, no expressions
  • One test result can link to multiple scenarios (tag collision)
Linking logic (abridged)
def link(scenarios, results):
    links = {}
    for scenario in scenarios:
        linked = [
            r for r in results
            if any(tag in scenario.tags for tag in r.tags)
        ]
        links[id(scenario)] = linked
    return links

Aggregator

The ReportAggregator transforms linked data into report views. All methods are static:

MethodOutput
build_views()Per-scenario, per-layer result groups with required-layer status
completion_stats()Total complete/incomplete counts with percentage
feature_breakdown()Per-feature completion stats
layer_stats()Per-layer test counts, durations, percentages, visual widths
failure_breakdown()Failures grouped by feature/scenario with stack traces
health_checks()All four health check statuses (progress, pyramid, E2E time, unlinked)
unlinked_results()Test results with no matching scenario

Renderer

The HtmlRenderer uses Jinja2 to produce a single self-contained HTML file. Everything is inlined:

  • CSS: Full stylesheet with light/dark themes via data-theme, falling back to prefers-color-scheme
  • JavaScript: Client-side hash routing (#/, #/pyramid, etc.), theme toggle (persisted in localStorage), sortable tables, search/filter
  • Logo: Optional base64-encoded PNG
  • Data: All report data is embedded as JSON in a <script> tag

The report file loads a monospace font from a CDN (Cascadia Mono via jsDelivr) but degrades gracefully to system ui-monospace if offline. No other external dependencies — it works on any OS in any modern browser.

JSON Report Model

report_model.build_report() takes the exact same views, stats, layer_stats, health_checks, and unlinked_results that HtmlRenderer.render() consumes and reshapes them into the dict defined by spectracer-report.schema.json. There is no second parsing or aggregation path — if it were, the HTML and JSON could disagree.

Key Transformations

  • Duration units — internal model stores seconds; the JSON schema expects milliseconds, so report_model.py converts on the way out.
  • Omit, don't null — duration and failureMessage are left out of a result entirely when the underlying JUnit/Cucumber data didn't provide them, rather than serialized as null.
  • Health rollup — the aggregator's four independent pass/warn/fail checks are reduced to a single worst-of summary.health.status (green/amber/red), with each non-passing check's message becoming a reasons[] entry. No separate health logic is reimplemented.
  • Relative feature paths — feature.file is computed relative to the config file's directory (via os.path.relpath, so it stays relative even outside that directory tree), never an absolute filesystem path — important for reports generated in CI and compared across machines.

Written to disk only when output_json is set in the config — omit it and cli.py skips this step entirely, leaving HTML-only behavior unchanged.

Data Flow

Here's exactly what happens when you run uv run spec-tracer (or spec-tracer after pip install):

  1. Config loading — cli.py discovers and reads the JSON config file. Validates required keys (features, output). unit, integration, and e2e are module-keyed objects ("" = unscoped).
  2. File collection — FileCollector (collectors.py) resolves literal file paths and recursively searches directories for .feature, .xml, and .json files, once per module key.
  3. Parsing — Each file type is parsed by its dedicated parser. Scenarios and test results are extracted into typed data classes; results carry the module key they were collected under.
  4. Linking — ResultLinker cross-references scenario tags against test result tags. OR logic: any shared tag creates a link.
  5. Aggregation — ReportAggregator computes all stats, views, breakdowns, and health checks from the linked data.
  6. Rendering — HtmlRenderer feeds the aggregated data into a Jinja2 template and writes the HTML output file.
  7. JSON output (optional) — if output_json is set, report_model.build_report() reshapes the same aggregated data into the schema-conformant JSON file.

Tag Resolution Order

When parsing a .feature file, tags are resolved as follows:

  1. All tags on lines immediately above a Scenario: line are collected
  2. Tags matching the pattern @require-(unit|integration|e2e)(:module)? are classified as layer requirements (optional :module suffix for all three layers)
  3. All remaining tags become linking tags
  4. If zero requirement tags were found, the scenario defaults to bare @require-e2e (unscoped)
  5. A module-scoped requirement is only satisfied by a linked result whose module matches that key; unscoped results never satisfy it
ℹ Feature-level tags are not inherited Tags placed on the Feature: line are ignored by SpecTracer. Only scenario-level tags participate in linking and requirement checking.

Testing Strategy (Dogfooding)

SpecTracer tests itself using its own input formats — an approach called outside-in dogfooding.

LayerFrameworkOutput FormatFeeds Tool As
E2EbehaveCucumber JSONe2e
UnitpytestJUnit XMLunit
IntegrationpytestJUnit XMLintegration

The CI pipeline runs all three layers, collects the outputs, and feeds them into SpecTracer to generate a self-report — a coverage report for SpecTracer itself. Every feature starts as a .feature file, validated by a behave E2E scenario before implementation.

Dogfooding CI pipeline (simplified)
uv run pytest tests/unit --junitxml=reports/unit.xml
uv run pytest tests/integration --junitxml=reports/int.xml

# E2E is split by module so @require-e2e:<module> can be dogfooded
uv run behave features/coverage_linking.feature --tags=-@FC-004 -f json -o reports/e2e-linker.json
uv run behave features/coverage_health.feature -f json -o reports/e2e-aggregator.json
uv run behave features/coverage_dashboard.feature -f json -o reports/e2e-renderers.json
uv run behave features/coverage_module_scope.feature -f json -o reports/e2e-parsers.json
# …plus collectors / report_model / edge splits (see .github/workflows/ci.yml)

uv run spec-tracer
# Output: reports/spectracer-report.html — a self-report
# Output: reports/spectracer-report.json — same self-report, machine-readable

SpecTracer's own scenarios use module-scoped tags such as @require-e2e:linker and @require-unit:parsers, and spectracer.config.json registers Cucumber JSON under matching module keys. Every self-report run also sets output_json. Both the integration test suite and the behave suite validate the generated JSON against spectracer-report.schema.json using jsonschema.Draft7Validator — the schema is enforced by the tool's own tests, not just documented.

✓ What gets measured gets managed SpecTracer's own self-report is the team's quality dashboard. If the tool's coverage drops, we fix it before shipping new features.
